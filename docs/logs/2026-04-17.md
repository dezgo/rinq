### 1. Bug fix: Cancel Transfer dropped the customer (`rinq/web/templates/phone.html`)

**Report:** Shristi — while warm-transferring to Rhae, Rhae couldn't hear her; when she tried to cancel the consult the customer got cut off too.

**Investigation:** Traced call `CAe626945c9654baa3b305374400a72278` in `journalctl -u rinq`. Timeline at 23:22:xx:

- 34.362 — Shristi's Cancel Transfer POST hits server (GET conference)
- 34.599 — DELETE consult participant (Rhae kicked)
- **34.989** — `/api/voice/call-ended` (from Shristi's browser) already running in a parallel worker
- **35.003** — `Call ended (browser signal): CAe626…` — customer's SID marked ended
- 35.073 — Cancel POST tries to unhold customer (too late; customer already gone)
- 35.485 — consult-status callback fires "`ended mid-consult … attempting rejoin`" (salvage, futile)
- 36.127 — participants=1 (Shristi alone)

The cancel itself wasn't the killer — a concurrent `/voice/call-ended` with the customer's SID was. Most likely source: `currentCall.on('disconnect')` at `phone.html:4840` firing when the browser's Twilio Device leg had a transient disconnect during the consult. That handler unconditionally calls `endCall()`, which reads `currentQueueCallerSid` (= customer's SID) and posts it to `/voice/call-ended`.

**Fix:** Guard at the top of `endCall()` (line 3266) — when `isConsulting` is true, null out `currentQueueCallerSid`, `currentTransferKey`, `isConsulting`, `consultCallSid`. The function then falls back to the agent's own browser leg SID via `currentCall.parameters.CallSid`. The customer stays held in the main conference; the server's existing consult-status callback handles unhold/rejoin.

**Commit:** `6f5377c`

## Key Discoveries

- **`endCall()` mid-consult was the real bug** (not the cancel flow itself). The cancel flow was doing its job — but another code path posted `/voice/call-ended` with the customer's SID concurrently. Documented as gotcha #32 in CLAUDE.md.
- **"Rhae couldn't hear Shristi" has no server-side cause.** Consult setup looks correct: customer held (`hold=True, hold_url`), Rhae joined `role=agent_no_exit`, Shristi never muted, `endConferenceOnExit=False` on all participants. Likely a one-off Twilio media path issue or a mic/permission hiccup on Shristi's browser. Told her to watch for recurrence.

## Key discoveries

- **REST-API-created calls to Client destinations drop `From`.** `client.calls.create(to='client:X', from_='+PSTN')` with a non-Twilio-owned number often results in `call.parameters.From` being empty on the receiving browser's Voice SDK. Fix pattern: server-side fallback via Twilio REST API by call_sid.

- **`close_stale_calls` is the actual completion path for SIP-answered customer calls.** Twilio's status callback does NOT fire reliably for these. `/api/active-calls` polling compares Twilio's in-progress list against our call_log and auto-completes anything finished. Any completion logic (talk_seconds, `_credit_conference_participants`, etc.) must be aware this path exists.

- **`#transferError` toast was invisible.** Element lives inside `.transfer-modal`, which has `display: none` when not `.active`. `startTransferStatusPolling` closes the modal before calling `showTransferError` — message existed in DOM but was never visible. Any inline-modal feedback that must persist after failure needs its own container outside the modal.

- **`'transferred'` is a successful answered outcome.** When warm/blind transfer completes, the main call_log row's status becomes `'transferred'` and agent_email is set to the recipient. All "did this agent answer" SQL must include it. 12 queries had this oversight in `call_log_mixin.py`.

- **PWA manifest name is baked in at install/pin time.** Changing `/manifest.json` doesn't retroactively rename existing pinned apps.

- **`friendly_name` data hygiene.** draftcal's friendly_name is "Watson Blinds"; surfaces as the target name in DND/transfer-failure messages ("Watson Blinds is on Do Not Disturb"). Probably other users have similar issues.

- **Shristi's "call dropped" reconstruction.** Consult call to Cid returned `busy` from Twilio in <1s (browser-side rejection), 2.5s of customer hold music during failure handling, then UI closed the modal with an invisible error. Customer was never disconnected — they were blind-transferred to Cid 30s later and talked for 193s.

## Decisions

- **Backed out "defer customer hold until consult answers"** (Option B from earlier plan). During ringing the agent's browser leg stays in the conference with the customer — skipping pre-hold would risk leaking consult audio. Pre-hold is the correct architectural choice; the audio blip is acceptable.

- **Dropped Calls/Missed split on inbound agent table.** With current attribution, Calls == Answered and Missed is perma-zero. Fake columns are worse than no columns.

- **Include `'transferred'` in ALL answered-count SQL**, not just reports — consistency across SLA buckets, aggregate stats, agent breakdowns.

- **Server-side DND check, not just UI rejection.** Refuses at the server before creating the consult call — faster, more informative, doesn't wake the target device at all.

- **Persistent call-notice banner, not a toast, for transfer failures.** Toast was being missed (8-second auto-hide, and in this case invisible entirely).
